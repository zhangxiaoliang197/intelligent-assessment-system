## 一、项目概述

### 1.1 项目背景

本系统是面向**作战场景**的智能评估平台，旨在为军事指挥、作战决策、效能评估等关键环节提供智能化支持。系统集成了知识库管理、本体模型构建、智能问答、指标分析、方案评估等多种能力，通过统一的数据底座实现跨系统的数据共享和协同工作。

### 1.2 项目目标

- **智能化评估**：基于RAG知识库和本体模型，提供智能化的作战评估能力
- **知识驱动**：构建统一的知识底座，实现知识的统一管理和共享
- **高效交互**：支持多模态交互（文本、文件、语音），提升用户体验
- **模块化设计**：各功能模块独立部署，灵活扩展

## 二、系统架构设计

### 2.1 整体架构

系统采用**前后端分离**架构，前端使用现代化的前端框架构建统一的门户页面，后端按照功能模块分为多个微服务：

- **AI服务层（Python）**：RAG知识库、本体模型、智能问答、指标分析、方案评估
- **业务服务层（Java）**：基础管理系统、数据库管理、配置管理
- **数据层**：统一数据底座，支撑所有上层应用

### 2.2 技术选型

#### 2.2.1 前端技术栈
- **框架**：React 18 / Vue 3（推荐Vue 3）
- **UI组件库**：Ant Design Vue / Element Plus
- **状态管理**：Pinia / Vuex
- **路由管理**：Vue Router
- **构建工具**：Vite
- **图表库**：ECharts（用于指标树状图、知识图谱展示）
- **语音交互**：Web Speech API

#### 2.2.2 后端技术栈 - AI服务层（Python）
- **Web框架**：FastAPI
- **RAG框架**：LangChain / RAGFlow
- **向量数据库**：Milvus / ChromaDB
- **大模型对接**：vLLM / 本地LLM（支持Ollama等）
- **本体模型**：Protégé API / Neo4j
- **知识图谱**：Neo4j图数据库
- **语音处理**：Whisper（语音识别）/ Edge TTS（语音合成）

#### 2.2.3 后端技术栈 - 业务服务层（Java）
- **核心框架**：Spring Boot 3.x
- **ORM框架**：MyBatis-Plus
- **数据库**：PostgreSQL / MySQL
- **消息队列**：RabbitMQ / Kafka
- **缓存系统**：Redis
- **任务调度**：Quartz

### 2.3 部署架构

#### 2.3.1 部署环境要求
- **运行环境**：Linux（CentOS 7+）
- **硬件要求**：
  - CPU：16核+
  - 内存：64GB+
  - 硬盘：2TB+
  - GPU：NVIDIA GPU（用于大模型推理）
- **网络**：内部私有网络，不连接外部互联网
- **大模型**：本地部署的大模型服务（如Qwen、ChatGLM等）

#### 2.3.2 容器化部署
- 使用Docker容器化所有服务
- 使用Docker Compose进行服务编排
- 支持离线安装和部署

## 四、数据架构设计

### 4.1 统一数据底座

统一数据底座是整个系统的核心，支撑所有上层应用的数据需求。

```
┌─────────────────────────────────────────────────────────────┐
│                     统一数据底座                            │
│  ┌─────────────┬─────────────┬─────────────┬──────────┐ │
│  │ RAG知识库    │  本体模型    │  业务数据    │ 元数据    │ │
│  │（向量数据）  │（图数据）    │（结构化数据）│（数据字典）│ │
│  └─────────────┴─────────────┴─────────────┴──────────┘ │
│                           │                               │
│              ┌────────────┴────────────┐                │
│              │      数据服务层           │                │
│              │  ┌─────────┴─────────┐    │                │
│              │  │  数据访问层API    │    │                │
│              │  └───────────────────┘    │                │
│              └─────────────────────────────┘                │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 数据存储方案

| 数据类型 | 存储方案 | 特点 |
|---------|---------|------|
| 结构化数据 | PostgreSQL/MySQL | 事务支持，查询能力强 |
| 向量数据 | Milvus/ChromaDB | 高效相似性检索 |
| 图数据 | Neo4j | 关系存储和查询 |
| 文档数据 | 文件系统+对象存储 | PDF、Word等原始文件 |
| 缓存数据 | Redis | 高性能缓存 |

### 4.3 数据流转

```
数据上传 → 知识解析 → 向量化存储 → 知识库检索
                        ↓
                  本体建模 → 图谱存储
                        ↓
              大模型推理 → 智能服务
```

## 五、接口设计

### 5.1 API网关

统一网关管理所有服务的API入口，负责：
- 请求路由
- 身份认证
- 权限控制
- 限流熔断
- 日志记录

### 5.2 核心接口

#### 5.2.1 知识库接口
- `POST /api/knowledge/upload` - 上传知识文件
- `POST /api/knowledge/parse` - 解析知识文档
- `GET /api/knowledge/search` - 检索知识
- `GET /api/knowledge/stats` - 获取知识库统计

#### 5.2.2 本体模型接口
- `POST /api/ontology/create` - 创建本体模型
- `POST /api/ontology/entity` - 添加实体
- `POST /api/ontology/relation` - 添加关系
- `GET /api/ontology/graph` - 获取图谱数据

#### 5.2.3 智能问答接口
- `POST /api/qa/chat` - 发送问答请求
- `GET /api/qa/history` - 获取问答历史
- `POST /api/qa/feedback` - 提交反馈

#### 5.2.4 指标分析接口
- `POST /api/indicator/analyze` - 分析指标
- `GET /api/indicator/tree` - 获取指标树
- `GET /api/indicator/detail` - 获取指标详情
- `GET /api/indicator/algorithm` - 获取指标算法

#### 5.2.5 基础管理接口
- `POST /api/admin/database/config` - 配置数据库
- `POST /api/admin/dataset/create` - 创建数据集
- `POST /api/admin/model/config` - 配置大模型
- `POST /api/admin/evaluation/scheme` - 创建评估方案

## 八、开发计划

### 8.1 设计原则

#### 8.1.1 微服务架构
系统采用**微服务架构**设计，每个系统和工具都是**独立的服务单元**，具备以下特点：

- **独立部署**：每个服务可以独立打包、部署和启动
- **独立配置**：每个服务拥有独立的配置文件，支持运行时配置更新
- **独立扩展**：根据负载情况，可以独立扩展某个服务而不影响其他服务
- **松耦合**：服务之间通过API接口通信，降低相互依赖
- **技术多样**：不同服务可以根据需求选择最适合的技术栈

#### 8.1.2 可配置性设计
每个服务支持丰富的配置项，包括：

- **服务配置**：端口号、日志级别、健康检查等
- **数据库配置**：连接池大小、超时时间、重连策略等
- **外部依赖配置**：大模型接口、数据源连接、消息队列等
- **功能开关**：启用/禁用特定功能模块
- **资源限制**：CPU、内存、并发数等资源限制

所有配置支持：
- **配置文件**：YAML/JSON格式的配置文件
- **环境变量**：支持通过环境变量覆盖配置
- **配置中心**：可选的集中配置管理（Consul/Apollo）
- **动态更新**：部分配置支持运行时动态更新
