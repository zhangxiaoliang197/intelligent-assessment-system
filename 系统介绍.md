# 智能评估系统 — 系统介绍

**版本**：V1.0 | **日期**：2026-07-09

---

## 一、产品定位与用户画像

### 1.1 一句话定位

面向作战指挥场景的 **"AI 分析师"**——将非结构化的分析需求，自动转化为结构化的数据库查询、指标计算与综合结论。

### 1.2 核心解决的问题

传统作战评估中，指挥官或参谋人员面临三个断层：

| 断层 | 传统方式 | 本系统解决 |
|------|----------|-----------|
| **需求→数据** | 人工理解需求 → 找表 → 手写 SQL，依赖 DBA | AI 自动选表 + Text-to-SQL，零 SQL 门槛 |
| **数据→洞察** | 导出 Excel → 人工制图制表 → 写报告，数小时 | 自动执行 + LLM 分析，秒级出结论 |
| **知识→决策** | 专业知识散落在文档/人脑中，无法系统化调用 | RAG 知识库统一管理，对话即检索 |

### 1.3 目标用户

| 角色 | 使用场景 | 核心诉求 |
|------|----------|----------|
| **指挥决策人员** | 快速了解战损/战果/消耗态势 | 自然语言提问，即刻获得数据+图表+结论 |
| **参谋/分析人员** | 构建指标体系、评估作战方案 | 知识库辅助 + LLM 推理，降低分析门槛 |
| **系统管理员** | 配置数据源、管理知识库、维护本体模型 | Web 界面操作，无需命令行 |

## 三、功能模块全景

系统按用户旅程分为三大类、七大模块：

```
         ┌────────────── 辅助系统（建基础） ──────────────┐
         │  知识库 (Knowledge)   本体模型 (Ontology)       │
         │  基础管理 (Admin)       数据源/指标/LLM 配置    │
         └──────────────────────┬──────────────────────────┘
                                │ 为上层提供数据底座
         ┌────────────── 分析工具（用起来） ──────────────┐
         │  智能问答 (QA)    指标分析 (Indicator)         │
         │  评估分析 (Evaluation)                          │
         └──────────────────────┬──────────────────────────┘
                                │
                         门户 (Portal)
                        统一入口 + 导航
```

### 3.1 智能问答（QA Service）

**用户心智模型**：像和专家对话一样提问，系统理解意图并给出答案。

```
用户: "上次对抗中红方战损最高的三个单位是哪些？"
  │
  ▼
RAG 检索知识库 → 找到相关条例/战报片段
  ↓
LLM 综合知识库片段 + 自身知识 → 生成回答
  ↓
流式返回，支持多轮追问
```

**特性**：
- 支持多轮对话，上下文窗口默认 5 轮
- 会话持久化，刷新页面后历史不丢失
- 知识库检索无结果时仍可依赖 LLM 自身知识回答
- 区分"知识库已知"与"大模型生成"两种来源

### 3.2 指标分析（Indicator Analysis）

**用户心智模型**：输入分析主题，自动生成分层级的指标体系树。

```
用户: "帮我分析一下空战效能"
  │
  ▼
调用 LLM 生成指标体系 JSON
  │  ├── 一级: 打击能力
  │  │   ├── 二级: 命中率 → 来源: admin-db（已在系统中定义）
  │  │   └── 二级: 摧毁率 → 来源: llm（大模型推断）
  │  ├── 一级: 生存能力
  │  └── 一级: 保障能力
  ▼
前端 ECharts 树状图展示，支持展开/折叠
```

**指标来源标注**：`admin-db`（数据库已配置） / `knowledge`（知识库已知） / `llm`（大模型推断），让用户知道每个指标的可靠性。

### 3.3 评估分析（Evaluation）— 多智能体协同

**用户心智模型**：选择数据源 → 输入分析需求 → 系统自动走完"理解需求 → 查数据 → 给结论"全流程。

**三种执行路径**（由 Orchestrator 自动路由）：

```
用户提问
  │
  ▼
Orchestrator 意图识别 (LLM)
  │
  ├── "最近一周的战损趋势"
  │     → data_query → 7 步标准流程
  │
  ├── "整体作战效能评估"
  │     → combat_effectiveness → 预配置 SQL 模板 + LLM 分析 + 综合结论
  │
  ├── "红蓝制空权对比"
  │     → air_superiority → 提取区域参数 + SQL 注入 + 红蓝对比
  │
  └── 无数据源的纯理论问答
        → general_analysis → 直接 LLM 回答
```

**智能降级**：
- 选了数据源但问题不需要查数据库 → 自动切回纯 LLM 模式
- 选了专项 Agent 但不匹配 → 自动降级为通用 data_query

### 3.4 知识库（Knowledge Base）

管理和维护系统的知识库，为所有智能服务提供知识支撑。
- 文档上传：支持 PDF/DOCX/TXT/MD/CSV
- 自动解析、分片、TF-IDF 向量化
- 分类管理、标签管理、统计信息
- JSON 文件持久化，原子写入 + .bak 备份

### 3.5 本体模型（Ontology Model）

构建和管理本体模型，建立概念之间的语义关系。
- 本体模型 CRUD / 实体 CRUD / 关系 CRUD
- 图谱数据聚合（nodes/links）
- 导入/导出本体模型 JSON
- 前端 ECharts 图谱可视化

### 3.6 基础管理（Admin System）

系统的配置中枢：
- **数据库配置**：连接管理 / 多数据源 / 连接测试 / 驱动动态加载
- **数据集管理**：创建数据集 / 表结构读取 / 字段标注 / 业务含义标注
- **指标管理**：CRUD + 数据集关联 + 字段映射 + 计算方法
- **大模型配置**：多配置管理 + 活跃配置切换（支持 deepseek/openai/vllm）
- **多数据库支持**：MySQL、PostgreSQL、Oracle、达梦 V8、SQL Server

## 四、多智能体架构深度解析

### 4.1 为什么需要多智能体？

| 单一 LLM 方案 | 多智能体方案 |
|---------------|-------------|
| 一次 Prompt 要包含选表+SQL+分析，Prompt 过长 → 幻觉率高 | 每个 Agent 只做一件事，Prompt 精准 → 可靠性高 |
| 新增分析场景 = 重写整套 Prompt | 新增场景 = 在 `queries.json` 加一条 SQL 模板 |
| 出错后无从排查 | 7 步可观测，每一步状态/结果独立上报前端 |

### 4.2 智能体组件

位于 `python/qa-service/agents/`：

| 文件 | 智能体 | 职责 |
|------|--------|------|
| `orchestrator.py` | 编排智能体 | 意图识别、路由选择、制定分析计划 |
| `workflow.py` | 工作流引擎 | 串联全部 7 步骤，管理状态流转 |
| `text_to_sql.py` | Text-to-SQL 智能体 | 根据表结构 + 问题生成 SELECT 语句 |
| `analyst.py` | 分析智能体 | 基于查询结果生成 2-3 条建议 |
| `combat_effectiveness_agent.py` | 作战效能智能体 | 使用预配置 SQL 模板，多维度分析汇总 |
| `air_superiority_agent.py` | 制空权分析智能体 | 提取区域参数 + 预配置 SQL，红蓝双方对比 |
| `state.py` | 共享状态 | 所有 Agent 共享的状态容器 |
| `tools.py` | 工具层 | HTTP 调用 admin-service 获取数据 / 执行 SQL |

### 4.3 通用 data_query 工作流（7 步）

```
步骤 1: Orchestrator 意图识别
         ↓ LLM 分析问题意图，提取实体/条件/维度/分析计划
步骤 2: 数据源探查
         ↓ 获取数据库所有表名（超时 8 秒，失败不中断）
步骤 3: 检查数据集 & 指标
         ↓ 从 admin-service 获取数据集定义和指标公式
步骤 4: 智能选表 + 读取表结构
         ↓ 打分机制精选最多 5 张表 + JDBC 读取列信息
步骤 5: Text-to-SQL 生成
         ↓ LLM 生成 SELECT → 安全校验（禁 DML/DDL）→ 分号清洗
步骤 6: SQL 执行
         ↓ JDBC 查询，获取原始数据
步骤 7: Analyst 分析建议
         ↓ LLM 基于数据生成 2-3 条可执行建议
```

### 4.4 智能选表打分机制（Step 4）

```
打分规则:
  - 分析计划中直接提到表名            → +100
  - 表名关键词匹配用户问题            → +20
  - 表名关键词匹配分析计划            → +10
  - 分析计划中出现"表 XXX"模式       → +200
  - 数据集关联的表优先保留
  - 最多选 5 张表（数据集表额外放宽到 6 张）
```

**产品价值**：避免给 LLM 塞入 50 张表的 Schema → 减少 Token 消耗 → 降低幻觉率 → 选错表的概率从 30% 降到个位数。

### 4.5 SQL 安全校验（Step 5）

生成 SQL 后经过三重校验：
1. 从 markdown 代码块或纯文本中提取 SQL
2. 安全检查：禁止 INSERT/UPDATE/DELETE/DROP/ALTER/TRUNCATE 等危险操作
3. 格式校验：必须以 SELECT 或 WITH 开头，括号匹配检查
4. 末尾清洗：去除分号和不可见控制字符（Oracle 兼容）
5. 失败时重试 1 次

### 4.6 专项 Agent 对比

| | 作战效能 Agent | 制空权 Agent |
|------|------|------|
| **配置来源** | `config/queries.json` | `config/air_queries.json` |
| **选 SQL 方式** | 关键词匹配 + LLM 二次筛选 | 提取区域 + LLM 筛选 |
| **SQL 注入** | 无 | `{region}` 占位符 → 区域参数替换 |
| **分析方式** | 逐条执行 + LLM 逐条分析 + 综合总结 | 逐条执行 + LLM 逐条分析 + 红蓝对比 |
| **可视化** | bar/line/pie/table | bar/line/pie/table |

## 五、技术架构决策与利弊

### 5.1 为什么是 Python AI + Java 业务双栈？

| 问题 | 决策 | 原因 |
|------|------|------|
| AI/LLM 开发用 Java 还是 Python？ | **Python** | FastAPI 异步性能好，LLM/LangChain 生态全在 Python |
| 数据库管理/CRUD 用 Python 还是 Java？ | **Java** | JDBC 多数据库兼容成熟，URLClassLoader 动态加载驱动是 Java 独有优势 |
| 两者如何通信？ | **HTTP REST** | 最简单，内网延迟可忽略 |

### 5.2 为什么不引入专用向量数据库？

PRD 中规划了 Milvus/ChromaDB，但实际实现用了 **sklearn TF-IDF + cosine_similarity**。

| 对比 | Milvus | TF-IDF（当前实现） |
|------|--------|-------------------|
| 检索精度 | 高（语义级） | 中等（关键词级） |
| 运维复杂度 | 高（需额外部署） | 零（Python 库内置） |
| 适用场景 | 海量文档语义检索 | 百/千级文档关键词检索 |

**决策逻辑**：V1.0 知识库文档量有限（百级），内网环境不允许随意安装中间件，优先保证可部署性。

### 5.3 为什么 JSON 文件而不是数据库？

知识库、本体模型、会话记录用 JSON 文件持久化：

| 考量 | 决策 |
|------|------|
| 读写模式 | 读多写少、单机够用、结构灵活 |
| 依赖控制 | 零依赖，内网部署无需额外数据库实例 |
| 数据安全 | 原子写入 + `.bak` 备份 |
| 迁移成本 | 后续迁移到 SQLite/MySQL 成本很低 |

### 5.4 当前版本核心取舍

| 取舍 | 选择 | 代价 |
|------|------|------|
| 向量检索 | TF-IDF（非语义检索） | 同义词/近义词检索弱 |
| 数据持久化 | JSON 文件 | 不支持并发写、数据量大时性能差 |
| 服务通信 | HTTP（非消息队列） | 同步耦合，一个服务慢会影响调用方 |
| LLM 调用 | urllib.request | 不支持自动重试、流控、连接池 |
| 部署 | Docker run 脚本（非 K8s） | 不支持自动扩缩容 |
| 前端 | Vue 3（无 SSR） | 首屏加载较慢（内网系统无此需求） |

## 九、产品设计亮点总结

| 维度 | 具体体现 |
|------|----------|
| **用户友好** | 自然语言提问，零 SQL 门槛；流式反馈不黑屏等待；步骤可视化增强信任感 |
| **可靠性** | SQL 安全校验 + 意图兜底降级 + 超时重试 + 失败不中断工作流 |
| **可扩展** | 新增分析场景只需配置 JSON，无需改代码；多数据库驱动动态加载 |
| **可部署** | 内网离线一键部署；无外部依赖（向量库/消息队列/图数据库全部用轻量替代） |
| **可观测** | 全链路 7 步状态上报；步骤级别的成功/失败标记；前端实时进度展示 |
| **知识管理** | RAG 检索 + 指标体系 + 本体模型三合一，知识可积累、可复用 |
| **领域深度** | 作战效能/制空权专项 Agent，不是简单的通用问答，而是深度领域定制 |
| **多智能体协同** | Orchestrator 编排 + 专项 Agent + 兜底降级，兼顾专业性与灵活性 |
| **Text-to-SQL 管线** | 智能选表 → LLM 生 SQL → 安全校验 → 执行 → 分析，全链路打通 |
| **指标来源透明** | admin-db / knowledge / llm 三种来源标注，用户知道每个指标的可靠性 |
